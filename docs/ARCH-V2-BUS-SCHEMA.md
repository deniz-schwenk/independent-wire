# V2 Pipeline Architecture — Bus Schema

**Status:** Architecture decision, S15 (2026-04-29). Pre-implementation.
**Replaces:** Ad-hoc Topic Package construction in `src/pipeline.py` and `src/pipeline_hydrated.py`.
**Migration approach:** Big-bang refactor. The current V1 pipeline is git-tagged as `v1-final` before V2 work starts, so V1 remains accessible as a rollback baseline if V2 implementation gets stuck.

---

## 1. Why this exists

The current pipelines build the final Topic Package by aggregating agent outputs in Python — each agent emits its own schema, pipeline code transforms and merges those outputs into a TP. Five classes of bugs trace back to this aggregation pattern, all surfaced in S14/S15:

- **ID-consistency bugs.** `rsrc-NNN` IDs leaked into the final TP because the renumbering helper (Fix-3) only normalised perspectives. Other TP fields (divergences, transparency, bias-analysis) carried unconverted IDs.
- **Aggregation duplications.** `bias_analysis.factual_divergences` was a Python copy of `qa_analysis.divergences`. Two locations, identical content, easy to drift.
- **Pre-research artefacts surviving downstream.** Coverage gaps generated by Phase 2 reducer described "no UK domestic outlets" while the final source pool included two UK outlets — pre-research statements that stale-d after research stages added sources.
- **Per-actor metadata grafted from the wrong dimension.** `actor.region` was set to the country of the source citing the actor. Trump cited in Handelsblatt → `region: "Germany"`. Semantically meaningless.
- **Editor-time selection_reason becoming false.** Editor decides at the start of the pipeline based on Curator-time source counts. By the time the TP is rendered, the selection reasoning often refers to source volumes that are no longer accurate.

All five bugs share a root: aggregation logic in pipeline code is the failure surface. Each new field needs explicit aggregation handling. Each new agent risks introducing or breaking aggregation invariants. The complexity is not in the domain — it is in the routing.

The V2 architecture removes the aggregation surface entirely. Agents write into named slots of a shared state object (the **Bus**); pipeline stages run in sequence, each operating on the Bus; rendering is selection from the Bus.

---

## 2. The Bus metaphor

A pipeline run is a fleet operation. One **RunBus** drives through the early stations carrying all passengers — RSS findings, curator clusters, editor assignments. At the topic-selection station, the pipeline runner dispatches **N TopicBuses** (one per produced TP). Each TopicBus carries the payload for exactly one Topic Package and drives the rest of the journey alone. At the terminus, each TopicBus is rendered into its TP output; the RunBus retains run-level state (run_id, date, total stages executed) for logs and per-run reports.

The metaphor is two-tiered, mirroring the pipeline's actual shape. There is one journey of the fleet (the run), and there are N parallel journeys of TPs after dispatch.

- The **RunBus** has a fixed seating plan with **run-scoped seats**: pipeline metadata, the upstream collection phases (curator findings, editor assignments), and at the end of the run a **manifest** of dispatched TopicBuses (status, identifiers, summary stats — not the full TopicBus contents).
- The **TopicBus** has a fixed seating plan with **topic-scoped seats**: the selected editorial assignment, research dossiers, perspective clusters, writer article, QA corrections, bias findings, render-output slots.
- The **Pipeline Runner** holds the collection of TopicBuses during topic-stage execution. The RunBus does not carry TopicBuses while they are being mutated. Only after all topic stages complete does `finalize_run` write a manifest into the RunBus.
- Each TopicBus carries a **read-only reference to the RunBus** for slots needed at render time (`run_id`, `date`, etc.). Topic-stages cannot mutate any RunBus slot.
- The seating plans (RunBus schema, TopicBus schema) are the single source of truth. Stages operate on whichever Bus type is declared in their interface.

This metaphor is not just a description — it is the structural commitment. The two seating plans together ARE the V2 schema, and they cleanly separate run-scoped state from topic-scoped state.

## 3. Architectural principles

### 3.1 Originary output, structurally enforced

Every slot has exactly one **owner** — the agent or pipeline stage authorised to write to it. Agents do not write to slots owned by other agents. Pipeline orchestration enforces this: the agent's emit is mapped onto its declared slot via Python, and other slots are not in scope of that emit.

This codifies the architect's "intelligent deterministic data routing" principle: each agent receives only what it needs, emits only what is originary to it. Aggregation, copying, derivation are pipeline concerns, not agent concerns.

### 3.2 Agents are isolated from the Bus Schema

An agent's prompt describes its input contract and its output contract in agent-local terms. The Curator does not know that its `topics[]` array becomes `run_bus.curator_topics`. The QA agent does not know that its `article` output becomes `topic_bus.qa_corrected_article`. The mapping is the pipeline's job.

This isolation has three consequences:
- Agents are reusable across pipeline variants (production, hydrated, future variants) without prompt changes.
- The Bus schemas can evolve (new slots, renamed slots) without touching agent prompts.
- Agents emit structures local to their task; the pipeline arranges those structures into the global state.

### 3.3 Empty-then-fill — the universal mirror pattern

Where an agent semantically "modifies" data produced earlier (QA corrects the Writer's article, Perspective-Sync adjusts cluster labels), the V2 architecture uses one consistent pattern for both modification and mirroring.

**The pattern:**

1. The modification target slot (e.g. `qa_corrected_article`) is initialised with an **empty default** — nested fields are empty strings, empty lists, empty dicts depending on type. The slot exists in the schema from TopicBus construction onward, but its content is empty until later.
2. The modifying agent emits **only the fields it actually changes**. Fields it does not change are emitted as empty (empty string, empty list, `null` — the agent's prompt declares the convention).
3. After the agent runs, a **deterministic mirror stage** walks the target slot and the source slot side-by-side. For every field that is empty in the target slot, it copies the value from the source slot.

**The result:**

- The target slot is always fully populated after the mirror stage runs.
- Render layers read the target slot directly without conditional fallback logic.
- Agents save tokens — they do not re-emit content they did not change.
- The pipeline's data routing is deterministic: empty means unchanged, non-empty means modified. No agent has to decide whether to pass-through.

**Schema metadata:**

Each modification target slot is annotated with a `mirrors_from` schema attribute pointing at its source slot. The mirror stage is generic — it reads `mirrors_from` from the schema, locates the source slot, and runs the empty-then-fill walk. No hard-coded mirror logic per slot.

| Modification target | Source slot | Set by `mirror_*` stage after which agent runs |
|---|---|---|
| `qa_corrected_article` | `writer_article` | after `qa_analyze` |
| `perspective_clusters_synced` | `perspective_clusters` | after `perspective_sync` (hydrated) or unchanged (production — `perspective_sync` does not run) |

**Definition of "empty":**

The mirror stage uses a deterministic `is_empty(value)` function:
- `None` → empty
- `""` (empty string) → empty
- `[]` (empty list) → empty
- `{}` (empty dict) → empty
- `0` → **not empty** (numeric zero is valid data)
- `False` → **not empty** (boolean false is valid data)

Slots that genuinely should remain empty in the target (rare; not currently any) would simply not carry a `mirrors_from` annotation — the mirror stage skips them.

**Agent-prompt implication:**

Agents that target mirror-pattern slots must have prompts that explicitly instruct empty-on-no-change emission. The QA prompt today emits the full corrected article unconditionally; this must be updated as part of V2 implementation to: *"emit only changed fields. When no correction applies to a field, leave it empty/null. The pipeline mirrors unchanged fields from the source slot after this stage."* Perspective-Sync today already follows this pattern correctly.

### 3.4 Granular slots, structured contents

Top-level slots in each Bus type are granular by phase or function — `qa_corrected_article` is one top-level TopicBus slot, distinct from the original `writer_article` slot. Within a slot, the structure follows the agent's natural output (a four-field article object stays a dict).

Granular at the top-level for clarity and provenance. Structured inside slots to avoid pointless flattening (no `qa_corrected_headline` separate from `qa_corrected_body` — they belong together as one corrected article).

### 3.5 Pre-initialisation

At RunBus construction (before any agent runs), all RunBus slots are present with typed empty defaults (lists `[]`, dicts `{}`, strings `""`, scalars `None`). Configuration values that come from CLI flags or environment (such as `max_produce`) are populated by the `init_run` stage at that moment.

At TopicBus construction (during the `instantiate_topic_buses` stage, after `select_topics`), all TopicBus slots are present with typed empty defaults except for slots that mirror inputs from the RunBus (`editor_selected_topic` is populated at construction time).

This guarantees that every consumer (later agent, deterministic stage, render layer) can rely on slot presence. No `dict.get(key, default)` patterns. No "is this slot already created?" checks. Strict-mode validation at Bus boundaries can use the master schema directly.

### 3.6 Render is selection

Both Bus types are internal pipeline state. What ships externally — TP JSON to disk, MCP server response, RSS feed, JSON-API payload, HTML render — is the output of a render function that selects from the relevant Bus or Buses.

A render function takes a TopicBus plus a read-only RunBus reference, selects from both, and returns the consumer-specific output. RSS picks `topic_bus.qa_corrected_article.headline` and `topic_bus.qa_corrected_article.summary`. The default TP-on-disk render picks all public-facing TopicBus slots plus `run_bus.run_id` for traceability.

Render functions are pure, parametrised by consumer, defined once. Adding a new output format means adding one render function — not changing the pipeline.

### 3.7 Visibility as schema-level metadata

Every slot in both Bus types carries a `visibility` attribute as **schema-level metadata** (a field on the Pydantic model, not just a documentation column). Visibility values: `tp` (public Topic Package), `mcp` (MCP server response), `rss` (RSS feed), `internal` (never exposed externally).

Render functions filter generically using the schema metadata: `topic_bus.fields_with_visibility("rss")` returns the RSS-eligible slots. Adding a new slot with a `visibility` annotation means it appears automatically in the matching render outputs — no per-render-function code change.

The tables in Section 4 list visibility values for documentation, but the Pydantic schema is authoritative. The tables and the schema must stay in sync, and where they differ, the schema is the source of truth.

### 3.8 Hierarchical Bus separation

The two-Bus architecture is the structural answer to a real shape in the pipeline: collection is run-scoped (one Curator-Editor sequence per pipeline run, regardless of how many TPs are produced), production is topic-scoped (each TP runs Researcher through Bias independently). Forcing both into one Bus would either duplicate the run-scoped data N-fold (one Bus per TP, all carrying identical Curator findings) or require ad-hoc shared-state hacks. Two Bus types are a small upfront cost that matches the domain shape.

The cost: stages must declare which Bus type they operate on. Mitigations: type hints, stage decorators that validate Bus type at registration, clear documentation of which slots live in which Bus.

## 4. The Bus schemas (granular, comprehensive)

This section enumerates every slot in both Bus types. Slots are grouped by phase. Each slot specifies:
- **Owner**: the stage or agent that writes to it. `init` means initialised by the Bus constructor.
- **Initial value**: what the slot holds at Bus-construction time.
- **Final value**: what the slot holds after its owner has executed.
- **Visibility**: which render layers expose it (`tp` = default TP-on-disk, `mcp` = MCP server, `rss` = RSS feed, `internal` = never exposed externally).
- **mirrors_from** (where applicable): the source slot for the empty-then-fill mirror pattern.

The tables are documentation. The Pydantic schema (to be implemented in `src/bus.py`) is the authoritative source.

### 4A. RunBus schema

The RunBus is created once per pipeline run and lives for the duration of the run. It carries run-scoped state that all TopicBuses can reference.

#### 4A.1 Pipeline metadata

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `run_id` | init | `""` | UUID-ish run identifier | `tp`, `mcp`, `internal` |
| `run_date` | init | `""` | ISO date string | `tp`, `mcp`, `internal` |
| `run_variant` | init | `""` | `"production"` or `"hydrated"` | `internal` |
| `max_produce` | init | `3` | The number of TopicBuses this run intends to dispatch. Read by `select_topics` (top-N selection) and by `finalize_run` (intent-vs-actual reporting). Set by `init_run` from the `--max-produce` CLI flag (default 3). | `internal` |
| `run_stage_log` | init | `[]` | List of `{stage, started_at, ended_at, status, scope}` per executed stage. `scope` is `"run"` or `"topic:{slug}"`. | `internal` |
| `run_topic_manifest` | init / `finalize_run` | `[]` | After all topic stages complete: list of `{topic_id, topic_slug, status, stages_completed}` summary entries — one per dispatched TopicBus. Does NOT carry the TopicBus contents themselves. | `internal` |

The Pipeline Runner holds the actual collection of TopicBus instances during topic-stage execution. The RunBus learns about that collection only at the end, via the manifest written by `finalize_run`.

#### 4A.2 Curator phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `curator_findings` | curator | `[]` | Raw RSS findings the Curator considered (input pool) | `internal` |
| `curator_topics_unsliced` | curator | `[]` | All topic clusters the Curator built, before any clipping | `internal` |
| `curator_topics` | curator | `[]` | Top-N clusters passed to Editor (typically 10) | `internal` |

#### 4A.3 Editor phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `editor_assignments` | editor | `[]` | Editor's prioritised list (10 entries: title, priority, selection_reason, follow_up_to, follow_up_reason). Python adds `id` and `topic_slug` post-emit. | `internal` |

`editor_assignments` is the run-scoped result of editorial decision-making. Topic-selection at the next stage chooses the top `max_produce` entries from this list and dispatches one TopicBus per selected entry.

### 4B. TopicBus schema

A TopicBus is created during the `instantiate_topic_buses` stage, one per topic the Editor selected for production. Each TopicBus represents the production of exactly one Topic Package.

Every TopicBus carries:
- A read-only reference to its parent RunBus (for `run_id`, `run_date`, etc. at render time).
- Its own state for the topic-scoped phases.

#### 4B.1 Topic identity

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `editor_selected_topic` | init (`instantiate_topic_buses` stage) | the assignment dict from `run_bus.editor_assignments` selected for this TopicBus | unchanged after init | `internal` |

`editor_selected_topic.id`, `.topic_slug`, `.priority` are the source of truth for TP-level metadata; later slots reference them.

#### 4B.2 Hydration phase (hydrated variant only)

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `hydration_urls` | init (extracted from `editor_selected_topic.raw_data`) | `[]` | URLs identified for full-text fetch | `internal` |
| `hydration_fetch_results` | hydration_fetcher | `[]` | Per-URL fetch results: `{url, status, extracted_text, outlet, country, language, error}` | `internal` |
| `hydration_phase1_analyses` | hydration_phase1_aggregator | `[]` | Per-article extraction: `{article_index, summary, actors_quoted[]}` | `internal` |
| `hydration_phase2_corpus` | hydration_phase2_reducer | `{preliminary_divergences: [], coverage_gaps: []}` | Cross-corpus reduction: divergences and gaps observed across the hydrated articles | `internal` |
| `hydration_pre_dossier` | init (built from phase1 + phase2) | `{sources: [], preliminary_divergences: [], coverage_gaps: []}` | Pre-research dossier (sources here carry IDs `hydrate-rsrc-NNN`) | `internal` |

In the production variant, all hydration slots remain at empty defaults and are never populated. Render layers never read them for production output.

#### 4B.3 Researcher phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `researcher_plan_queries` | researcher_plan | `[]` | Multilingual search queries the Plan agent produced | `internal` |
| `researcher_search_results` | researcher (web_search) | `[]` | Raw search results | `internal` |
| `researcher_assemble_dossier` | researcher_assemble | `{sources: [], preliminary_divergences: [], coverage_gaps: []}` | Web-search-built dossier (sources here carry IDs `research-rsrc-NNN`) | `internal` |

#### 4B.4 Source merge and renumbering (deterministic stages)

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `merged_sources_pre_renumber` | init (Python merge stage) | `[]` | `hydration_pre_dossier.sources` + `researcher_assemble_dossier.sources` concatenated, IDs still in `hydrate-rsrc-NNN` / `research-rsrc-NNN` form | `internal` |
| `final_sources` | init (Python renumber stage) | `[]` | Same sources after final ID assignment: `src-001`, `src-002`, ..., `src-NNN`. This is the canonical source list referenced everywhere downstream and rendered in the TP. | `tp`, `mcp` |
| `id_rename_map` | init (Python renumber stage) | `{}` | Mapping from pre-renumber IDs (both `hydrate-rsrc-NNN` and `research-rsrc-NNN`) to final `src-NNN` IDs. Used by the universal ID-normalisation stage that runs immediately after to clean up pre-research-side text fields. | `internal` |
| `merged_preliminary_divergences` | init (merge stage) | `[]` | Combined from both pre-research dossiers, IDs already normalised to `src-NNN` via `id_rename_map` | `internal` |
| `merged_coverage_gaps` | init (merge stage) | `[]` | Combined gap statements from both pre-research dossiers (post-research validation runs later) | `internal` |

The renumbering stage is the single point where IDs are finalised. Every later agent receives source records and source-id references already in `src-NNN` form. No agent ever sees a `rsrc-` token after this stage.

#### 4B.5 Perspective phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `perspective_clusters` | perspective | `[]` | Position clusters: `[{id: "pc-NNN", position_label, position_summary, source_ids[], actors[]}]`. Source-ids reference `final_sources` (already normalised). Cluster IDs (`pc-NNN`) are emitted by the agent. | `internal` |
| `perspective_missing_positions` | perspective | `[]` | Stakeholder voices the dossier could not source | `internal` |

#### 4B.6 Writer phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `writer_article` | writer | `{headline: "", subheadline: "", body: "", summary: ""}` | Writer's draft article. Citations in body use `[src-NNN]` referencing `final_sources`. | `internal` |

#### 4B.7 QA+Fix phase (empty-then-fill mirror)

| Slot | Owner | Initial | Final | Visibility | mirrors_from |
|---|---|---|---|---|---|
| `qa_problems_found` | qa_analyze | `[]` | List of factual problems QA identified | `tp`, `mcp` | — |
| `qa_proposed_corrections` | qa_analyze | `[]` | One-liner correction per problem, in same order as `qa_problems_found` | `tp`, `mcp` | — |
| `qa_corrected_article` | qa_analyze + `mirror_qa_corrected` stage | `{headline: "", subheadline: "", body: "", summary: ""}` | After QA emits: fields the agent corrected hold the corrected text; fields the agent did not change are empty. After the mirror stage: empty fields are filled from `writer_article`. Final state always has all four fields populated. | `tp`, `mcp`, `rss` | `writer_article` |
| `qa_divergences` | qa_analyze | `[]` | Cross-source factual disagreements. References to source IDs are already in `src-NNN` form because QA reads from `final_sources`. | `tp`, `mcp` | — |

Mirror semantics: the QA agent is instructed to emit only changed fields and leave unchanged fields empty. The `mirror_qa_corrected` stage runs after `qa_analyze` and walks the four fields of `qa_corrected_article`. For each empty field, it copies the value from `writer_article`. Final result: a complete corrected article with QA's modifications where applicable and Writer's content elsewhere.

#### 4B.8 Perspective-Sync phase (hydrated variant)

| Slot | Owner | Initial | Final | Visibility | mirrors_from |
|---|---|---|---|---|---|
| `perspective_clusters_synced` | perspective_sync + `mirror_perspective_synced` stage | `[]` | After Perspective-Sync emits: a list of cluster delta objects (cluster id plus changed `position_label` and/or `position_summary` only). After the mirror stage: full cluster list with deltas applied where present, original cluster fields elsewhere. | `tp`, `mcp` | `perspective_clusters` |

In the production variant, `perspective_sync` does not run. The mirror stage still runs and produces `perspective_clusters_synced` as a 1:1 copy of `perspective_clusters` — render layers always read `perspective_clusters_synced`, never the unsynced version, regardless of variant.

#### 4B.9 Bias Detector phase

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `bias_language_findings` | bias_language | `[]` | Linguistic bias findings: `[{excerpt, issue, explanation}]` | `tp`, `mcp` |
| `bias_reader_note` | bias_language | `""` | One-paragraph reader-facing note | `tp`, `mcp` |

The Bias Language Detector reads the corrected article (`qa_corrected_article`) and emits only what is originary to it: linguistic findings and reader-note. Other bias dimensions (source, geographical, selection, framing) are derived from other slots at render time — see Section 4B.12.

#### 4B.10 Coverage gaps (post-research validation)

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `coverage_gaps_validated` | init (Python validation stage, runs after `final_sources` and `source_balance` exist) | `[]` | `merged_coverage_gaps` filtered against `source_balance` to drop gap statements that the final source pool empirically refutes ("no X-language sources" when X is present, etc.) | `tp`, `mcp` |

The unvalidated `merged_coverage_gaps` remains in the TopicBus for full provenance. Render layers reference `coverage_gaps_validated` only.

#### 4B.11 Source balance and rendered transparency

| Slot | Owner | Initial | Final | Visibility |
|---|---|---|---|---|
| `source_balance` | init (Python aggregation, runs after `final_sources`) | `{by_country: {}, by_language: {}, represented: [], missing_from_dossier: []}` | Computed from `final_sources` | `tp`, `mcp` |
| `transparency_card` | init (Python aggregation, runs after QA) | `{}` | Computed from TopicBus state. Contains: `selection_reason` (cleaned via stale-quantifier strip, references `editor_selected_topic.selection_reason`), `pipeline_run` (`run_id`, `date` — pulled from the parent RunBus), `article_original` (mirror of `writer_article` if QA changed anything, else absent), `qa_problems_found`, `qa_proposed_corrections`. | `tp`, `mcp` |

#### 4B.12 Bias Card (rendered, multi-slot derived view)

The Bias Card is not a TopicBus slot. It is a **derived render output** that pulls from five existing slots to surface the five Vision-promised bias dimensions. The render function is `compose_bias_card(topic_bus)`; its output appears in the public TP under the key `bias_analysis`.

Mapping of Vision dimensions to TopicBus slots:

| Vision dimension | Sourced from |
|---|---|
| **Language** | `bias_language_findings` (LLM-emitted linguistic findings) |
| **Source** | `source_balance.by_country`, `source_balance.by_language`, `source_balance.represented` |
| **Geographical** | `source_balance.by_country` (regional view subset) |
| **Selection** | `coverage_gaps_validated`, `perspective_missing_positions`, `qa_problems_found` |
| **Framing** | `perspective_clusters_synced` (cluster-level position framings), `qa_divergences` (cross-source narrative differences) |

The composed `bias_analysis` block in the rendered TP looks like:

```json
"bias_analysis": {
  "language": [...findings from bias_language_findings...],
  "source": {...subset of source_balance...},
  "geographical": {...regional breakdown of source_balance.by_country...},
  "selection": {
    "coverage_gaps": [...from coverage_gaps_validated...],
    "missing_positions": [...from perspective_missing_positions...],
    "qa_findings": [...from qa_problems_found...]
  },
  "framing": {
    "position_clusters_summary": [...high-level summary of perspective_clusters_synced...],
    "cross_source_divergences": [...from qa_divergences...]
  },
  "reader_note": "..."  // from bias_reader_note
}
```

This is the single place where the five Vision dimensions are surfaced explicitly to TP consumers. Adding new dimensions later (or reorganising them) is a change to `compose_bias_card` — no schema change needed.

## 5. Pipeline stages (V2)

The V2 pipeline is a sequence of stages, each operating on either the RunBus or a single TopicBus. Stages declare their target Bus type explicitly. Each stage has a precondition (which slots must be populated before it runs) and a postcondition (which slots it populates).

### 5.1 Stage list (production variant)

```
RUN STAGES (operate on RunBus, run once per pipeline run)
1. init_run                      — Construct RunBus with all slots at empty defaults; set run_id, run_date, run_variant="production"; populate max_produce from the --max-produce CLI flag (default 3)
2. curator                       — Reads RSS feeds; populates curator_findings, curator_topics_unsliced, curator_topics
3. editor                        — Reads curator_topics + previous_coverage; populates editor_assignments
4. select_topics                 — Python: sort editor_assignments by priority desc, take top run_bus.max_produce; produces a list of selected assignments
5. instantiate_topic_buses       — Python: for each selected assignment, the Pipeline Runner constructs a TopicBus with that assignment as editor_selected_topic; the Runner holds the collection (the RunBus does not)

TOPIC STAGES (operate on each TopicBus, one execution per TopicBus)
6. researcher_plan               — Reads editor_selected_topic; populates researcher_plan_queries
7. researcher_search             — Python: executes queries via web_search; populates researcher_search_results
8. researcher_assemble           — Reads researcher_search_results; populates researcher_assemble_dossier
9. merge_sources                 — Python: hydration_pre_dossier is empty in production; copies research dossier sources into merged_sources_pre_renumber
10. renumber_sources              — Python: assigns final src-NNN IDs; populates final_sources, id_rename_map
11. normalize_pre_research        — Python: rewrites IDs in merged_preliminary_divergences and merged_coverage_gaps using id_rename_map
12. perspective                   — Reads final_sources + merged_preliminary_divergences + merged_coverage_gaps; populates perspective_clusters, perspective_missing_positions
13. mirror_perspective_synced     — Python: empty-then-fill mirror. perspective_clusters_synced is empty (production never runs perspective_sync); the stage fills it from perspective_clusters as a 1:1 copy.
14. writer                        — Reads final_sources + perspective_clusters_synced + perspective_missing_positions + merged_coverage_gaps + editor_selected_topic; populates writer_article
15. qa_analyze                    — Reads writer_article + final_sources + perspective_clusters_synced; populates qa_problems_found, qa_proposed_corrections, qa_corrected_article (only changed fields; empty otherwise), qa_divergences
16. mirror_qa_corrected           — Python: empty-then-fill mirror. Fills empty fields of qa_corrected_article from writer_article. Final state: qa_corrected_article has all four fields populated.
17. compute_source_balance        — Python: aggregates final_sources into source_balance
18. validate_coverage_gaps        — Python: filters merged_coverage_gaps against source_balance; populates coverage_gaps_validated
19. bias_language                 — Reads qa_corrected_article; populates bias_language_findings, bias_reader_note
20. compose_transparency_card     — Python: assembles transparency_card from editor_selected_topic.selection_reason (cleaned), the parent RunBus's run_id and run_date, qa_problems_found, qa_proposed_corrections, and writer_article (as article_original) if qa modified anything

RUN STAGES (operate on RunBus again, run once after all topic stages complete)
21. render                        — For each TopicBus the Runner holds, for each requested output (tp, mcp, rss): apply the corresponding render function with (topic_bus, run_bus); write to disk or return
22. finalize_run                  — Python: write run_stage_log, write run_topic_manifest into run_bus (summary entries, not full TopicBus contents), close RunBus
```

### 5.2 Stage list (hydrated variant)

Run stages 1–5 identical to production. Topic stages diverge after `instantiate_topic_buses`:

```
RUN STAGES — same as production stages 1–5

TOPIC STAGES (operate on each TopicBus)
6. attach_hydration_urls          — Python: extracts hydration_urls from editor_selected_topic.raw_data (Curator-side cluster matching happens here)
7. hydration_fetch                — Python: aiohttp + trafilatura fetch of hydration_urls; populates hydration_fetch_results
8. hydration_phase1               — Reads hydration_fetch_results (success-only filter); populates hydration_phase1_analyses
9. hydration_phase2               — Reads hydration_phase1_analyses + hydration_fetch_results metadata; populates hydration_phase2_corpus
10. assemble_hydration_dossier    — Python: combines phase1 + phase2 + fetch metadata into hydration_pre_dossier (sources carry hydrate-rsrc-NNN IDs)
11. researcher_plan               — Reads editor_selected_topic + hydration_pre_dossier (gap-aware queries); populates researcher_plan_queries
12. researcher_search             — Python: executes queries; populates researcher_search_results
13. researcher_assemble           — Reads researcher_search_results; populates researcher_assemble_dossier (sources carry research-rsrc-NNN IDs)
14. merge_sources                 — Python: concatenates hydration_pre_dossier.sources and researcher_assemble_dossier.sources; populates merged_sources_pre_renumber
15. renumber_sources              — Python: assigns final src-NNN IDs; populates final_sources, id_rename_map
16. normalize_pre_research        — Python: rewrites IDs in merged_preliminary_divergences and merged_coverage_gaps
17. perspective                   — Same as production stage 12
18. writer                        — Same as production stage 14
19. qa_analyze                    — Same as production stage 15
20. mirror_qa_corrected           — Same as production stage 16
21. perspective_sync              — Reads perspective_clusters + qa_corrected_article + qa_problems_found + qa_proposed_corrections; emits cluster deltas (only changed fields) into perspective_clusters_synced
22. mirror_perspective_synced     — Python: empty-then-fill mirror. For clusters where perspective_sync emitted deltas, those are kept; for clusters not in the delta output, the stage copies from perspective_clusters as 1:1.
23. compute_source_balance        — Same as production stage 17
24. validate_coverage_gaps        — Same as production stage 18
25. bias_language                 — Same as production stage 19
26. compose_transparency_card     — Same as production stage 20

RUN STAGES — same as production stages 21–22
```

The hydrated variant adds five hydration topic stages and one perspective-sync topic stage. All other stages are identical and reuse the same agent calls.

Note: in production, `mirror_perspective_synced` runs immediately after `perspective` because there is no `perspective_sync` stage. In hydrated, `mirror_perspective_synced` runs after `perspective_sync` so the mirror can fill from `perspective_clusters` only the clusters that the sync agent did not deliver deltas for.

### 5.3 Stage interface

Every stage is a typed function. Two distinct interfaces:

```python
async def run_stage(run_bus: RunBus) -> RunBus:
    """Execute a run-scoped stage. Reads from declared input slots, writes to declared output slots."""
    # ...
    return run_bus

async def topic_stage(topic_bus: TopicBus, run_bus: RunBus) -> TopicBus:
    """Execute a topic-scoped stage on a single TopicBus.
    The run_bus is read-only — used for run_id, run_date, and other run-scoped reads.
    Topic stages must not mutate run_bus.
    """
    # ...
    return topic_bus
```

Stages declare their preconditions (input slots) and postconditions (output slots) as decorators or class attributes. The pipeline runner validates preconditions before each stage runs and validates postconditions after — a stage that fails to populate its declared output slots is a hard error, not a silent partial result. The runner also enforces the read-only nature of `run_bus` in topic stages by passing a Pydantic frozen-model proxy that raises on attempted mutation.

### 5.4 Topic-stage execution model

Topic stages run **sequentially per TopicBus**, but TopicBuses themselves can run **independently in parallel** — there is no cross-TopicBus dependency. The V2 runner can choose to execute topic stages serially across TopicBuses (current V1 behaviour, simpler to debug, easier to budget) or in parallel (faster wall-clock, more concurrent LLM load).

For V2 launch, topic-stage execution is **serial per TopicBus** (one TopicBus completes its full topic-stage sequence before the next starts). Parallelisation is a follow-up performance workstream once the architecture is stable.

## 6. Render layer

The render layer reads from a TopicBus (and its parent RunBus reference) and produces output formats.

### 6.1 Render functions

Each render function selects from the relevant Bus or Buses and may apply consumer-specific formatting:

```python
def render_tp_public(topic_bus: TopicBus, run_bus: RunBus) -> dict: ...
def render_mcp_response(topic_bus: TopicBus, run_bus: RunBus) -> dict: ...
def render_rss_entry(topic_bus: TopicBus, run_bus: RunBus) -> dict: ...
def render_internal_debug(topic_bus: TopicBus, run_bus: RunBus) -> dict: ...   # everything, no filtering
def compose_bias_card(topic_bus: TopicBus) -> dict: ...                        # multi-slot derived view (Section 4B.12)
```

`run_bus` is read-only for all render functions — render is selection, not mutation.

### 6.2 Default `tp` render (production-grade Topic Package on disk)

The on-disk Topic Package JSON is built by `render_tp_public(topic_bus, run_bus)`. Selected slots:

- `id`: from `topic_bus.editor_selected_topic.id`
- `metadata`: `{title, date, status, topic_slug, priority, follow_up}` from `topic_bus.editor_selected_topic` and `run_bus.run_date`
- `article`: from `topic_bus.qa_corrected_article` (always; mirror semantics guarantee completeness)
- `sources`: from `topic_bus.final_sources`, with internal-prefix fields stripped
- `perspectives`: `{position_clusters: from topic_bus.perspective_clusters_synced, missing_positions: from topic_bus.perspective_missing_positions}`
- `divergences`: from `topic_bus.qa_divergences`
- `gaps`: from `topic_bus.coverage_gaps_validated`
- `bias_analysis`: from `compose_bias_card(topic_bus)` — the multi-slot derived view per Section 4B.12
- `transparency`: from `topic_bus.transparency_card`
- `source_balance`: from `topic_bus.source_balance`

Slots NOT in the public TP render: every internal slot, all `researcher_*` (process state), all `hydration_*` (process state), `merged_*` (intermediate), `id_rename_map`, `writer_article` (the uncorrected version — replaced by `qa_corrected_article` for the public render).

The selection is implemented by filtering against the `visibility` schema metadata (Section 3.7), not by hand-listing slot names. The list above is documentary.

### 6.3 RSS render (example)

```python
def render_rss_entry(topic_bus: TopicBus, run_bus: RunBus) -> dict:
    return {
        "title": topic_bus.qa_corrected_article["headline"],
        "description": topic_bus.qa_corrected_article["summary"],
        "link": f"https://independentwire.org/{topic_bus.editor_selected_topic['topic_slug']}",
        "pubDate": run_bus.run_date,
        "guid": topic_bus.editor_selected_topic["id"],
    }
```

### 6.4 MCP render

Returns a structured dict suitable for MCP consumption — by default identical to `render_tp_public` plus optional access to `topic_bus.qa_problems_found` and `topic_bus.qa_proposed_corrections` for clients that want to consume the QA reasoning chain.

### 6.5 Consumer choice is consumer choice

Both Buses carry everything. What a specific output exposes is determined by the render function for that output, plus the schema-level visibility metadata. New requirements ("expose the merged_preliminary_divergences in MCP for clients building their own bias analysis") are render-only changes — no pipeline rewiring, no agent prompt changes.

## 7. ID normalisation invariant

The TopicBus enforces a strict ID-state invariant by stage:

- **Before `renumber_sources`** (topic stage 10 production / 15 hydrated): IDs in `hydration_pre_dossier.sources`, `researcher_assemble_dossier.sources`, and any agent-emitted source-id references use the agent-local form (`hydrate-rsrc-NNN`, `research-rsrc-NNN`).
- **After `renumber_sources`**: `final_sources` carries `src-NNN`. `id_rename_map` maps every pre-renumber ID to its post-renumber ID.
- **`normalize_pre_research`** (topic stage 11 / 16): rewrites IDs in `merged_preliminary_divergences` and `merged_coverage_gaps` using `id_rename_map`.
- **All later stages** (perspective onward): receive only `src-NNN` references. Agents emit references using `src-NNN` because that is what the pipeline gives them as input.

Result: no agent post-Perspective ever sees a `rsrc-` token. The TP cannot leak `rsrc-` IDs because no slot containing them is rendered, and the slots that are rendered have already been normalised.

The S15-era `_normalize_all_source_ids` recursive sweep becomes unnecessary — the architecture makes its purpose structurally redundant.

## 8. Migration plan (big-bang with Git-tag rollback)

### 8.1 Approach

V1 pipeline code (`src/pipeline.py` and `src/pipeline_hydrated.py`) is replaced in one workstream with V2 implementation. Before the workstream starts, the current finalised V1 codebase is committed and pushed to `origin/main`, then tagged as `v1-final`. If V2 implementation gets stuck, V1 can be pulled back from the tag at any time without losing progress.

The two pipelines are folded into one generic pipeline runner that takes a stage list (production or hydrated) as input. Variant-specific behaviour lives in stage definitions, not in subclassed Pipeline classes.

### 8.2 Implementation order

1. **Tag V1 final.** `git tag v1-final && git push --tags`. This is the rollback baseline.
2. **Define the Bus classes.** Pydantic models with all slots, default values, type annotations, visibility metadata, mirrors_from metadata. Two source files: `src/bus.py` (RunBus + TopicBus + read-only proxy logic).
3. **Define the master Bus schemas.** Used for boundary validation, not for in-flight slot writes (those are typed Python).
4. **Define stage interface.** Function signatures for `run_stage` and `topic_stage`, precondition/postcondition declaration mechanism, validation, read-only proxy enforcement.
5. **Port deterministic stages.** `init_run`, `select_topics`, `instantiate_topic_buses`, `merge_sources`, `renumber_sources`, `normalize_pre_research`, `compute_source_balance`, `validate_coverage_gaps`, `compose_transparency_card`, `mirror_qa_corrected`, `mirror_perspective_synced`, `attach_hydration_urls`, `assemble_hydration_dossier`, `finalize_run`. Plus the generic `mirror_stage` helper that reads `mirrors_from` schema metadata. These are pure Python and the easiest to test offline.
6. **Update agent prompts where required.** QA prompt today emits the full corrected article unconditionally; update to empty-then-fill convention ("emit only changed fields, leave unchanged empty/null"). Other agents already follow the V2 convention or do not need it. Update is a precondition for the V2 pipeline to work as designed.
7. **Anglicise agent and slot names.** Rename `agents/perspektiv/` → `agents/perspective/`, `agents/perspektiv_sync/` → `agents/perspective_sync/`. Update all references in code, schemas, prompts, and documentation. This is the moment.
8. **Port agent stages.** Each agent gets a thin wrapper stage that knows how to map agent output to its declared TopicBus (or RunBus, for curator and editor) slots. Agent prompts have been updated in step 6 where needed.
9. **Port render functions.** `render_tp_public`, `render_mcp_response`, `render_rss_entry`, `render_internal_debug`, `compose_bias_card`. All take `(topic_bus, run_bus)`. Visibility-based filtering is generic.
10. **Port the pipeline runner.** Replaces `Pipeline.run`, `Pipeline.run_partial`, `PipelineHydrated`. One generic runner that takes a stage list, holds the TopicBus collection during topic-stage execution, and orchestrates RunBus + TopicBus passing.
11. **Update `scripts/run.py`.** Switch to V2 runner. `--hydrated` flag selects hydrated stage list instead of production stage list. Add `--max-produce N` flag (default 3) — passed into `init_run` and stored in `run_bus.max_produce`.
12. **Migrate tests.** V1-specific tests are rewritten or replaced with V2 tests. Add property-based tests for Bus invariants: every slot pre-initialised, every modification target has a `mirrors_from` annotation, no orphan slot writes, visibility metadata consistent with render-function selections.
13. **Smoke validation.** Production smoke + hydrated smoke. Compare structurally (slot presence, source-count, cluster-count, no rsrc-leaks, no stale gaps) against pre-V2 reference TPs taken from S15 runs. Acceptable diffs: presentation refinements (cleaner metadata, no leaked rsrc-IDs, no stale gaps, richer bias-analysis block). Unacceptable diffs: missing fields, structurally different article content, regressions in source-id consistency.

### 8.3 What gets thrown away

- `src/pipeline.py:_normalize_all_source_ids` — superseded by stage `renumber_sources` invariant
- `src/pipeline.py:_validate_coverage_gaps` — kept, but moved into the `validate_coverage_gaps` stage
- `src/pipeline.py:_strip_stale_quantifiers` — kept, but moved into `compose_transparency_card`
- `src/pipeline_hydrated.py:_convert_rsrc_to_src_in_perspectives` (Fix-3) — superseded by stage `normalize_pre_research` and the structural invariant

- The German agent name `perspektiv` — replaced by English `perspective` consistently across code, prompts, and folders

### 8.4 What stays

## 9. Out of scope for V2

Not part of the V2 refactor — to be addressed in subsequent workstreams or evaluated empirically:

- **Stakeholder-Gap-Auditor agent.** A future agent that examines the corrected article and identifies relevant actor types not covered. Plugs into the V2 architecture as a new topic-stage and new TopicBus slot.
- **Editorial-drift-note transparency field.** Tracks when Editor's pre-research selection_reason is overtaken by the actual coverage. Plugs in as a new transparency slot.
- **QA+Fix splitting.** Whether to split QA into a fact-check agent and a divergence-detection agent. Empirical decision based on Phase D/E eval results.
- **Production-hardening for hydrated.** Bot-blocking resilience, browser-based fetcher, multi-URL fallback per source cluster. Independent of Bus architecture.
- **Topic-stage parallelisation.** Running multiple TopicBuses concurrently rather than serially. Architecturally trivial in V2 but deferred for stability.
- **WP-OPUS-4.7-MIGRATION.** Model-tier change — orthogonal.
- **WP-BUDGET-LIMITS / WP-OUTPUT-SAFETY.** Production safety rails — orthogonal.
- **Render-layer extensions.** Additional output formats (PDF, structured email digest, etc.) — easily added once render functions are first-class.
- **Configuration layer.** A general-purpose config file (`config/pipeline.yaml` or similar) — for V2, runtime parameters like `max_produce` come from CLI flags only. A formal config layer can be added later as an additive concern; new parameters become individual RunBus slots until a config layer is justified.
- **Crash-recovery / per-stage snapshots.** Snapshot-per-stage as JSONL append on disk would enable resume-after-crash and richer postmortem debugging. Cheap to add, deferred for V2 launch focus.

## 10. Decision log

- **2026-04-29 S15:** Architecture decision taken. Granular slots (Variante B2 from internal discussion). Empty-then-fill mirror pattern as the universal modification pattern (Mirror-Stages run after the modifying agent, fill empty fields from the source slot). Maximum slot inventory regardless of variant. Big-bang migration with `v1-final` git tag as rollback. Hierarchical Bus separation (Variante C: RunBus + TopicBus) — the pipeline's natural shape is a single run dispatching N topic-productions, the architecture mirrors that shape directly. TopicBuses live in the Pipeline Runner during execution (not in the RunBus); RunBus learns about them only at the end via `run_topic_manifest`.
- **2026-04-29 S15:** `max_produce` modelled as a RunBus slot, populated by `init_run` from the `--max-produce` CLI flag (default 3). Future configuration-file support is additive and out of scope for V2.
- **2026-04-29 S15:** Bias Card modelled as a multi-slot derived render output (Section 4B.12), not a single TopicBus slot. The five Vision dimensions (language, source, geographical, selection, framing) are explicitly mapped to existing slots and surfaced via `compose_bias_card`. This makes the Vision-promised five dimensions visible in the public TP without inventing new state.
- **2026-04-29 S15:** Visibility modelled as Pydantic field-metadata on the schema. Render functions filter generically by visibility tag rather than hand-listing slot names. Documentation tables are explanatory; the schema is authoritative.
- **2026-04-29 S15:** Anglicise the agent name `perspektiv` to `perspective` consistently — agent folder, code references, schema slot names, prompts, and documentation. Big-bang is the right moment for this naming consolidation.
- Architect's reasoning: aggregation surface in V1 is the failure source for five S14/S15 bugs. Fixing the bugs at the structural level (one RunBus, N TopicBuses, one stage list, one render layer, one universal mirror pattern) is preferred over patching aggregation logic.
